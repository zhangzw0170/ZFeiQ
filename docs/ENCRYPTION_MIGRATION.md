- 前向保密（PFS）：使用短期（ephemeral）X25519 密钥对可在握手后丢弃私钥，降低长期密钥泄露带来的历史消息泄露风险。
- 性能与资源消耗：在嵌入式 ARM（RK3566/RK3588 等）与低功耗设备上，X25519 与 ChaCha20‑Poly1305 的速度和能耗优势明显，能显著提高响应与降低 CPU 占用。
- 报文紧凑：椭圆曲线公钥（32B 原始）与 AEAD 载荷比 RSA 包裹更节省带宽，适合局域网与受限网络场景。
- 现代化算法栈：Curve25519/ChaCha20 是当前主流轻量、安全选项，便于后续使用硬件加速或更广泛的库支持。
- 简化握手开销：一次 ECDH 握手后复用对称密钥，避免逐条消息进行昂贵的非对称运算。


## 已注意到的安全与兼容权衡

- 认证缺口：当前握手流程交换 ephemeral 公钥并通过广播指纹 (`fp`) 检测对端公钥变化，但并没有用长期身份密钥对 ephemeral 公钥进行签名。因此在首次交互或不可信网络中仍有 MITM 风险。
- SID 与 nonce 设计：SID 为 8 字节（SHA256(key) 的前 8 字节），nonce 派生取 sha256(sid || label || ctr)[:12]。虽实现简洁，但 SID 长度和 nonce 派生策略需要在审计时确认是否满足目标威胁模型。
- 向后兼容性：若需同时与仍使用 RSA 的旧客户端互通，需要保留旧 `ENC;...` 解密路径或在握手阶段提供回退选项。


## 建议的改进方向（下一步）

1. 增加握手认证（优先级：高）
   - 使用长期身份密钥对 ephemeral 公钥进行签名。例如：引入 Ed25519（或用现有长期密钥签名格式）对 `pubA/pubB` 做签名字段，并在 `KX1/KX2` 中携带签名；接收方验证签名以抵抗 MITM。
   - 参考位置：`NZFeiQ/core/engine.py` 中 `identity_priv` / `identity_pub_bytes`，可用来实现签名或将长期身份替换为签名键对。

2. 审查并扩展 SID/nonce 方案（优先级：中）
   - 将 SID 扩展到 16 字节以降低碰撞概率；改进 nonce 派生以包含更多上下文（例如会话生成时间戳或更长的 SID 片段）。
   - 在文档中明确 nonce/ctr 的容错与恢复策略，避免因 ctr 回退导致 nonce 重用。

3. 完善重放与窗口管理（优先级：中）
   - 评估 `recv_window` 的滑动策略（当前保留最多约 1024 条），在高丢包/长断线场景下防止集合误增长或误判。

4. 兼容性与回退策略（优先级：中）
   - 在握手协商阶段加入 `ver` 与能力协商字段（已有 `ver=2` 与 `fp`），明确回退到 RSA‑ENC 的条件并实现双向能力探测。

5. 测试与 CI 覆盖（优先级：高）
   - 为握手流程增加单元/集成测试：模拟 KX1→KX2→ENCREADY、异常 tag、错误 SID、重放攻击等用例。
   - 在 CI 中把 `cryptography` 列为必装依赖并运行加密冒烟测试。

6. 安全审计清单（优先级：高）
   - 列出 threat model 条目：首次建立 MITM、长期身份泄露、nonce 重用、SID 碰撞、重放窗口溢出。为每项制定检测与缓解措施。


## 结语

该迁移是基于安全性（PFS）、性能（嵌入式友好）与运维成本（更少的每条消息非对称运算）三方面的权衡所得。若项目需要对外发布或在不可信网络中部署，建议优先完成“握手签名/认证”与“测试覆盖”的工作。有关实现细节与现有代码位置，请参考：

- `NZFeiQ/core/crypto.py`
